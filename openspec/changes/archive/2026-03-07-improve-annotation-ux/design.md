## Context

付箋（Sticky Notes）は、データモデルに対して非公式なメタデータを追加するために導入されました。しかし、現在の実装では追加時の視覚的なフィードバックが乏しく、他のモデリングオブジェクト（テーブルやドメイン）との統一感が欠けています。また、追加時に詳細パネルが表示されることで、キャンバス上での直接編集の邪魔になるというフィードバックがあります。

## Goals / Non-Goals

**Goals:**
- 付箋追加時のアニメーションを `TableNode` や `DomainNode` と同期させ、統一感を持たせる。
- 付箋を単独で追加した際に、カメラが自動的にその位置を捉えるようにする（自動フォーカス）。
- 付箋追加時に詳細パネルを自動的に開かないようにし、キャンバス上での編集に集中できるようにする。

**Non-Goals:**
- 付箋以外のオブジェクトの追加フローの変更。
- 詳細パネルの機能自体の変更。

## Decisions

### 1. `AnnotationNode` への `isNew` ステートとアニメーションの追加
`TableNode.tsx` と同様の `isNew` ステートを `AnnotationNode.tsx` にも導入し、追加から1秒間 `animate-creation` CSSクラスを付与します。
- **Rationale**: `index.css` に定義済みの `animate-creation` キーフレームアニメーションを流用することで、統一されたビジュアル体験を最小限のコード変更で提供できます。

### 2. `addAnnotation` におけるフォーカス制御の改善
`useStore.ts` の `addAnnotation` アクションを更新し、ターゲット（`targetId`）が指定されていない場合のみ `setFocusNodeId(newId)` を呼び出すようにします。
- **Rationale**: ターゲットがある場合はすでにそのオブジェクトに視線が向いているため、カメラ移動は不要です。ターゲットがない場合はどこに追加されたか不明瞭になりやすいため、フォーカスが必要です。

### 3. 詳細パネル表示の抑制
`toggleAnnotationSelection` や `addAnnotation` 実行時に、詳細パネルが自動で開かないようにフラグやアクションを制御します。
- **Rationale**: 付箋は追加直後にそのままキャンバス上の `textarea` で編集されるのが一般的なワークフローであり、画面を占有する詳細パネルを出すことはノイズになります。

## Risks / Trade-offs

- **[Risk] フォーカス時の混乱** → カメラ移動が激しすぎるとユーザーが酔ったり現在位置を見失ったりする可能性がある。
  - **Mitigation**: すでに `fitView` や `setFocusNodeId` は他の箇所で使用されており、その適切なアニメーション時間（800msなど）を維持することで混乱を最小限に抑えます。
- **[Risk] 詳細パネルが開かないことによる不便さ** → 色の変更などを詳細パネルで行いたいユーザーが、追加直後に操作できず戸惑う可能性がある。
  - **Mitigation**: 詳細パネル自体が完全に無効になるわけではなく、明示的にクリック（選択）し直すことで表示可能です。
