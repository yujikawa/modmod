## Context

現在のRightPanelにはTables・PathFinder・NoteSearchの3タブがある。いずれもテーブル単位の操作であり、カラムを横断的に検索する手段がない。Zustandストアはすでにパース済みのモデルデータ（テーブル・カラム情報）を保持しており、追加のデータ取得は不要。

## Goals / Non-Goals

**Goals:**
- RightPanelに `ColumnSearchTab` を追加し、カラム横断検索を実現する
- 検索対象：テーブルの `name` / `logical_name` / `physical_name` / `conceptual.description`、カラムの `logical.name` / `physical.name` / `logical.description` / BEAMタグ
- 検索結果をカラム単位のリストで表示（テーブル名は概念名大・論理名中・物理名小の階層表示）
- 結果クリックでキャンバス上の該当テーブルにフォーカス
- テキストは長い場合 `...` で切り詰め

**Non-Goals:**
- ファジー検索・形態素解析などの高度な検索アルゴリズム（シンプルな文字列 `includes()` で十分）
- WASM等のパフォーマンス最適化（対象データ規模では不要）
- カラム単体へのフォーカス（テーブル単位でのフォーカスで十分）
- 検索結果のソート・フィルタ機能

## Decisions

### 1. 検索アルゴリズム：インメモリ文字列検索（`includes()`）
- **採用理由**: 対象データはZustandストアのインメモリオブジェクト。現実的なデータモデルのカラム数は数百〜数千程度であり、`String.includes()` で体感ゼロ。
- **不採用の代替**: Fuse.js等のファジー検索ライブラリ、WASM。いずれも導入コストがベネフィットを上回る。

### 2. 検索インデックスの構築タイミング：クエリ入力ごとに都度計算
- **採用理由**: モデルデータはユーザー編集で随時変わるため、事前インデックスを持つとキャッシュ整合性の管理が必要になる。データ規模が小さいため都度計算で問題なし。

### 3. コンポーネント配置：`RightPanel/InformationSearchTab.tsx` として独立ファイル
- **採用理由**: 既存タブ（`TablesTab`・`PathFinderTab`・`NoteSearchTab`）と同じ規約に従い、RightPanel内に配置。

### 4. フォーカス実装：既存の `focusOnNode` / `selectTable` アクションを再利用

- **採用理由**: PathFinderTabなど既存タブが同様のフォーカス処理を実装済み。同じストアアクションを使うことで一貫性を保つ。

## Risks / Trade-offs

- **[リスク] カラム数が非常に多いモデルでのレンダリング負荷** → 検索結果の表示上限（例: 50件）を設けることで対応
- **[トレードオフ] フォーカスはテーブル単位** → カラム個別へのスクロールは対応しないが、DetailPanelでカラム一覧が確認できるため実用上の問題は小さい
