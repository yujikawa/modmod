## Context

現在のデータモデルは規模が大きくなると、特定のテーブル間の依存関係を追うのが困難になります。また、モデル全体に散らばった付箋の管理も不十分です。本設計では、これらを「検索可能」かつ「可視化可能」な情報として右パネルに統合します。

## Goals / Non-Goals

**Goals:**
- ER関係とLineageを統合した最短経路探索エンジンを実装する。
- 検索結果をキャンバス上で視覚的に強調（ハイライト）する。
- 全ての付箋をテキスト検索し、該当箇所へ即座に移動できる。
- 右パネルを機能ごとに整理されたタブ構成にする。

**Non-Goals:**
- パス検索における重み付け（特定のコスト計算）の実装（今回は単純なホップ数で計算）。
- プレゼンテーションモードとの統合（今回は独立して機能させる）。

## Decisions

### 1. グラフ探索エンジン (`lib/graph.ts`)
- **データ構造**: `schema` を走査し、ノード ID をキーとする隣接リストを生成します。
- **エッジの定義**: 
    - `relationships`: 1ホップ（無向として扱う）。
    - `lineage`: Upstream/Downstream 各1ホップ。
- **アルゴリズム**: 単純な最短経路（Shortest Path）を見つけるために BFS (Breadth-First Search) を使用します。

### 2. パスハイライトのステート管理
`useStore.ts` に以下の状態を追加します：
- `activeRightPanelTab`: 'tables' | 'path' | 'notes'
- `pathFinderResult`: `{ nodeIds: string[], edgeIds: string[] } | null`
- **Rationale**: 検索結果をグローバルステートに持つことで、`App.tsx` 内の `ReactFlow` コンポーネントが即座にレンダリングを調整できるようにします。

### 3. 右パネルのリデザイン
`RightPanel.tsx` を以下のサブコンポーネントに分割します：
- `EntitiesTab`: 現在のテーブル一覧。
- `PathFinderTab`: From/To 選択 ＋ 検索実行 ＋ パス一覧。
- `NoteSearchTab`: 全文検索 ＋ 付箋リスト。

### 4. 視覚的演出（スポットライト）
パスが選択された際、`App.tsx` のノード/エッジ生成ロジックで以下を適用します：
- パス外の要素: `opacity: 0.1`, `pointerEvents: 'none'`。
- パス内の要素: 境界線を太くし、`zIndex` を最前面へ。

## Risks / Trade-offs

- **[Risk] パフォーマンスの低下** → 超大規模モデル（1000テーブル超）でのグラフ探索。
  - **Mitigation**: 探索をバックグラウンド（Web Worker等）で行うか、あるいは探索深度に上限（例：最大10ホップ）を設けます。
- **[Risk] UIの複雑化** → 右パネルに機能が詰め込まれすぎる。
  - **Mitigation**: タブをアイコン化し、必要最小限のコントロールのみを表示するクリーンなデザインを維持します。
