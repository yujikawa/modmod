## Context

現在の Modscape 開発サーバー (`src/dev.js`) は Express を使用しており、モデルファイルの変更は `chokidar` で監視していますが、その変更をブラウザ側に通知する手段がありません。ユーザーは手動でリロードを行う必要があり、特に大きなモデルファイルを編集している際の確認作業が非効率です。

## Goals / Non-Goals

**Goals:**
- 外部エディタでのファイル保存から 1 秒以内にブラウザ側の表示が更新されること。
- ファイルの追加・削除も自動的に検知し、モデルリストを更新すること。
- 手動リロードボタンを廃止し、UI を簡素化すること。
- 大規模な YAML ファイルでも UI のフリーズを最小限に抑えて更新すること。

**Non-Goals:**
- 差分更新（モデルの一部だけを更新する）。
- 複数のユーザー間でのリアルタイム共同編集。

## Decisions

### 1. 通信プロトコル: WebSocket (ws ライブラリ)
**理由:** 低遅延であり、双方向通信が可能なため。追加のポートを開放せず、既存の Express サーバーと同じポートを使用します。

### 2. イベント駆動の包括的監視 (Chokidar `all` イベント)
**理由:** 現在は `change` のみ監視していますが、`add` (新規ファイル追加) や `unlink` (ファイル削除) にも対応させます。
- **ファイルイベントの分類:** 
    - `change`: 現在表示中のモデルが変更された場合に `refreshModelData` を実行。
    - `add`, `unlink`: ファイルリスト (`/api/files`) が変更された可能性があるため、リストを再取得。
- **デバウンス:** 一度に複数のイベントが発行された場合の負荷を防ぐため、サーバー側で 200ms のデバウンスを導入します。

### 3. UI の簡素化: 手動リロードボタンの削除
**理由:** リアルタイム同期が有効であれば、手動でのリロードは不要になります。ユーザーの混乱を避けるため、サイドバーヘッダーやパネル内のリフレッシュ用アイコンボタン (`RefreshCw`) を削除します。代わりに「Saving...」や「Saved」というステータス表示のみを残します。

## Risks / Trade-offs

- **[Risk]** ディレクトリ内の大量のファイル増減。
  - **Mitigation:** YAML (.yaml, .yml) ファイル以外の変更は無視し、監視対象を限定します。
- **[Risk]** 手動リフレッシュができなくなる不安感。
  - **Mitigation:** サーバー側での同期が確実に動作することを担保し、ブラウザ側の WebSocket 接続が切れた場合には再接続を試行していることをユーザーに伝えるインジケータ表示（必要に応じて）を検討します。
