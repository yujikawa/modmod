## Context

ModscapeはCLIサーバーとReactクライアントがWebSocketで通信するSPAです。これをテストするには、実際にCLIを起動した状態でブラウザ操作を行う必要があります。

## Goals / Non-Goals

**Goals:**
- コマンド一つで全主要機能の健全性が確認できること。
- テスト実行用のダミーYAMLファイルを用いて、実行環境に左右されない安定したテストを行うこと。
- UI崩れを検知するためのベースライン画像比較を導入すること。

**Non-Goals:**
- 100%のコードカバレッジ。まずは「重要なボタンや同期が壊れていないか」というハッピーパスの検証を優先します。
- 全てのブラウザ（Safari等）の網羅。まずは Chromium (Chrome) での安定動作を優先します。

## Decisions

### 1. ツール選定: Playwright
**理由:** 高速で安定しており、Auto-wait（要素が出るまで待つ）機能が強力なため。また、スクリーンショット比較やトレース表示が標準で備わっており、Modscapeのような複雑なUIのデバッグに適しています。

### 2. テスト用データの固定化 (`tests/fixtures/`)
**理由:** `samples/` のファイルはユーザーが書き換える可能性があるため、テスト専用の `test-model.yaml` を用意します。テスト実行前にこのファイルを一時ディレクトリにコピーして使用します。

### 3. Page Object Pattern の採用
**理由:** 「キャンバス」「サイドバー」「詳細パネル」など、コンポーネントごとに操作ロジックを分離することで、UI変更時のテスト修正コストを下げます。

## Risks / Trade-offs

- **[Risk]** WebSocket接続の待機。
  - **Mitigation:** テストコード内で、WebSocketが接続完了して `Live` バッジが出るまで待機するユーティリティを作成します。
- **[Risk]** ドラッグ＆ドロップの不安定さ。
  - **Mitigation:** React Flow のノード位置を固定座標で指定するなどの工夫をします。
- **[Risk]** テスト実行時間。
  - **Mitigation:** 開発時は特定のテストのみを実行できるようタグ付けを行います。
